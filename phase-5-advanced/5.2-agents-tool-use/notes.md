# 5.2 AI Agents and Tool Use

## What Is an Agent?

A simple LLM call is stateless: prompt in, response out. An **agent** adds a loop. The model is given
tools (functions it can call), and it iterates — calling tools, observing results, reasoning about what
to do next — until it decides it has a final answer.

```
Task
  │
  ▼
[Observe current state]
  │
  ▼
[Think: what should I do next?]  ← LLM
  │
  ▼
[Act: call a tool / produce output]
  │
  ├── if tool call → execute tool → inject result → loop back to Observe
  │
  └── if final answer → return to user
```

This loop is the core of every agent framework, whether it's LangChain, AutoGen, LangGraph, or a
custom implementation.

---

## The ReAct Pattern

ReAct (Reasoning + Acting) is the most widely used agent prompting pattern. Published by Yao et al.
(2023), it asks the model to interleave explicit reasoning steps with tool actions:

```
Thought: I need to find the current price of AAPL stock.
Action: search_web
Action Input: {"query": "AAPL stock price today"}
Observation: AAPL is currently trading at $189.45.

Thought: Now I can answer the question.
Final Answer: Apple (AAPL) is currently trading at $189.45.
```

Each iteration produces:
- **Thought** — the model's internal reasoning (not shown to the user)
- **Action** — which tool to call
- **Action Input** — parameters for the tool (usually JSON)
- **Observation** — the tool's return value (injected by your code, not generated by the LLM)

The model sees all previous Thought/Action/Observation turns in its context, allowing it to build
on previous results.

## Tool Definitions

Tools are described to the model in the system prompt or via a structured tool-calling API. Each tool
needs:

1. **Name** — short, descriptive, snake_case. The model uses this to call the tool.
2. **Description** — natural language explanation of what the tool does and when to use it.
   This is the most important field. A good description prevents wrong tool selection.
3. **Parameters** — JSON Schema defining input types, required fields, and field descriptions.

Example tool definition:

```json
{
  "name": "search_web",
  "description": "Search the internet for current information. Use when you need facts
                  that may have changed since your training cutoff, or need to verify
                  specific data.",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "The search query. Be specific and concise."
      }
    },
    "required": ["query"]
  }
}
```

The model generates a call like: `{"name": "search_web", "arguments": {"query": "Apple earnings Q3 2025"}}`

Your code then executes `search_web("Apple earnings Q3 2025")` and returns the result as an Observation.

## Tool Execution Loop

Your agent framework is responsible for:

1. Formatting tools into the prompt
2. Sending to the LLM
3. Parsing the LLM output (extract action name + input)
4. Looking up and calling the matching tool function
5. Injecting the result back into the conversation
6. Repeating until the model returns a final answer or max iterations is hit

The LLM never directly runs code — it only outputs text that describes what tool to call with what
arguments. **Your code** runs the tool. This separation is important for safety.

## LangGraph

LangGraph (from LangChain) models agents as state machines:

- **Nodes** — processing steps (call LLM, execute tool, format result)
- **Edges** — transitions between nodes, can be conditional
- **State** — a typed dictionary passed through the graph at each step
- **Checkpointing** — save state to a database, allowing pause/resume and debugging

LangGraph is powerful for complex multi-step agents but adds significant framework overhead.
For simple agents, a plain Python loop is cleaner.

```python
from langgraph.graph import StateGraph

graph = StateGraph(AgentState)
graph.add_node("llm_call", call_llm)
graph.add_node("tool_exec", execute_tool)
graph.add_conditional_edges("llm_call", should_continue, {
    "tool": "tool_exec",
    "end": END,
})
graph.add_edge("tool_exec", "llm_call")
```

## AutoGen: Multi-Agent Conversations

AutoGen (Microsoft) enables multiple agents to collaborate. Rather than one LLM calling tools,
you have multiple specialized agents that message each other:

- **UserProxy** — represents the human; routes messages, confirms actions
- **AssistantAgent** — the main LLM, generates plans and code
- **Specialized agents** — code executor, critic, domain expert

AutoGen is useful for complex tasks that benefit from debate and review between agents, but can be
expensive (many LLM calls) and hard to debug.

## Agent Memory

Agents need memory to be useful across interactions:

**Short-term memory (in-context):**
- The current conversation history
- Limited by context window (e.g., 200k tokens for Claude)
- Everything in the context is "remembered" perfectly

**Long-term memory (external):**
- Vector store of past conversation summaries or key facts
- Retrieved via similarity search at the start of each session
- Allows the agent to "remember" facts from weeks ago
- Tradeoff: retrieval is imperfect; important facts may not be retrieved

**Episodic memory:**
- Store complete past task executions
- Retrieve similar past tasks as few-shot examples
- Helps the agent learn from experience without fine-tuning

## Human-in-the-Loop

Not all agent actions should execute automatically. Dangerous or irreversible actions should pause
and request human approval:

```python
def execute_action(action, input_data):
    if action.risk_level == "high":
        print(f"Agent wants to: {action.name}({input_data})")
        confirmation = input("Approve? (y/n): ")
        if confirmation != "y":
            return "Action cancelled by user."
    return action.fn(**input_data)
```

Common patterns:
- Always approve before sending emails or making purchases
- Auto-approve read-only operations (search, read_file)
- Require approval for writes (delete_file, post_to_social, send_message)

## Safety Constraints

Agents operating in loops can go wrong in ways that single LLM calls cannot:

1. **Max iterations** — Always set a hard limit (e.g., 15 steps). Without it, a confused agent can
   loop indefinitely.
2. **Tool budget** — Limit expensive tool calls (e.g., max 5 web searches per task).
3. **Dangerous action detection** — Blocklist tools or patterns for destructive operations.
4. **Blast radius limitation** — Limit what the agent can access: only read files in `./sandbox/`,
   only query a read-only DB, only send emails to a pre-approved list.
5. **Timeout** — Kill the entire agent if total wall-clock time exceeds a threshold.

## Evaluation

Evaluating agents is harder than evaluating single-turn LLM calls:

- **Task completion rate** — Did the agent achieve the goal? (binary or graded)
- **Step efficiency** — How many steps did it take vs the minimum possible?
- **Tool selection accuracy** — Did it choose the right tools?
- **Trace analysis** — Review full Thought/Action/Observation traces for failure patterns
- **Human evaluation** — For open-ended tasks, human raters assess quality

Use tools like LangSmith or Langfuse to capture and review agent traces. A single confusing
trace can reveal a systematic failure mode (e.g., the agent always tries the wrong tool first
for a certain class of question).

---

## Common Failure Modes

| Failure Mode | Symptoms | Fix |
|---|---|---|
| Infinite loop | Agent keeps calling same tool with same input | `max_iterations` limit |
| Wrong tool selection | Agent uses `calculate` to search for facts | Improve tool descriptions |
| Hallucinated parameters | Agent passes non-existent keys to a tool | Validate against JSON schema |
| Context window exceeded | Long tool outputs push off earlier context | Summarize or truncate observations |
| Premature final answer | Agent answers without gathering enough info | Add explicit multi-step system prompt |
| Over-cautious / refuses | Agent won't act, keeps asking for clarification | Adjust system prompt |
